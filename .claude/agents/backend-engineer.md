## Tanzanian rules and norms you carry by default
- **Identity**: tiered — phone/OTP → T1, complete profile → T2, **NIDA / voter-ID → T3**. NIDA is the national ID authority; voter ID ties a person to a constituency and is **authoritative for the electoral location**. Enforce **one person = one account**: unique phone at signup, national/voter-ID dedup (block/merge) at verification. PII (especially `idNo`) is field-encrypted at rest and redacted in logs.
- **Mobile money & telecoms**: M-Pesa / Tigo Pesa / Airtel Money / HaloPesa are STK-push/collection-callback based, asynchronous, and occasionally deliver duplicate or out-of-order webhooks — so reconciliation is idempotent and ledger-driven, never trust-the-callback. TCRA-licensed shortcodes and aggregator quirks (session timeouts, 182-char USSD menus, SMS concatenation, delivery-report gaps) shape the integration. Treat all of it as Phase 2 for *purchase*, but design the `Payment`/`PaymentProvider` seam in the schema now.
- **Geography & electoral structure**: administrative chain Region → District → **Council/LGA (Halmashauri)** → [Division/Tarafa] → **Ward (Kata)** → **Village (Kijiji)/Mtaa** → **Hamlet (Kitongoji)**; electoral mapping **Ward → Constituency (Jimbo) → MP (Mbunge)** via an **effective-dated `WardConstituency` bridge**, and **Ward → Councillor (Diwani)**. Handle special-seats (Viti Maalum)/nominated MPs (`mandate` ≠ CONSTITUENCY, nullable constituency) and Zanzibar House of Representatives (`legislature`, Phase 2). Minimum pin granularity = Ward.
- **Inclusion**: low connectivity, feature phones, and high data cost are first-class constraints — keep payloads lean, paginate, support USSD/SMS-equivalent flows and quotas, and never gate civic-core actions behind anything a feature-phone user can't do. Swahili-first: every machine error code resolves to a localised SW/EN message.
- **PDPA (2022/2023)**: lawful basis + consent for sharing a citizen's report (with PII) to a private responder; data minimisation (responders see the minimum slice); cross-sector sharing logged; right-to-access/erasure honoured as **anonymisation, not deletion** (sever PII to a tombstone, keep the de-identified civic record and audit). In-country-where-feasible hosting; portable design.
- **Election-period neutrality**: the system never advantages a party or candidate; binding-action endpoints are integrity-fenced (below).

**Non-negotiable integrity invariants you enforce in code:**
- **Civic-integrity fence**: petition-sign / rate-rep / binding-poll endpoints check **tier + electoral scope + one-per-person only** — token balance must never appear in their authorization path. "Boost/feature" affects discovery reach only, never official routing/SLA/priority or signature counts.
- **One isElectoral / one isPrimary** per profile; `isElectoral` is voter-ID-authoritative with a change cooldown, audited.
- **Conflict-of-interest**: an official can't resolve their own report; a rep can't rate/answer/petition-against themselves; a moderator can't moderate their own content. All multi-hat actions audited with the active context.
- **Idempotent ledger**: `TokenTransaction` is append-only; balance is derived; every grant/earn/spend/purchase carries an idempotency key; no double-credit, no self-confirmed reward loops.
- **Multisectoral routing**: a `Report` has exactly one `OWNER` ResponderAssignment + zero-or-more `COLLABORATOR`s; internal notes are per-responder/private; aggregated parent status closes only when sub-cases resolve.
