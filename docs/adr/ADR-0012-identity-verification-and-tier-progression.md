# ADR-0012: Identity verification & tier-progression (T1‚ÜíT2‚ÜíT3) ‚Äî profile completion, pluggable ID verification, the operator (Moderator) queue, electoral-location authority, and staff TOTP

**Status:** Accepted ¬∑ 2026-06-23 ¬∑ Solution Architect (David Okello)
**Extends:** ADR-0011 (authentication & tokens ‚Äî live tier resolution, scope seam, JWT hardening, append-only audit, OTP machinery). ADR-0011 built the auth behaviour and the `TierService` that *computes* T0‚ÄìT3; this ADR builds the *path that earns* T2 and T3 and the first staff endpoint that consumes it.
**Grounding:** PRD ¬ß6.4 (D11‚ÄìD16), ¬ß7 (tiers T0‚ÄìT3), ¬ß9.0 (multi-location, primary vs electoral, ward‚Üíconstituency), ¬ß18 (security/verification), ¬ß24 (responder scope), ¬ß25.4 (`isElectoral` edge cases) / ¬ß25.5 (downgrade); D-Q1, D-Q2, D13, D15, D16. ## Context
The auth increment delivered a citizen at **T1** (phone+OTP) and a live `TierService` that *would* return **T2** once the profile is complete and **T3** once `profile.idVerified` is true ‚Äî but **nothing yet earns those flags**. There is no profile-completion contact-verify, no ID-submit, no operator decision, no electoral-location authority binding, and the verification queue has no staff endpoint. Building that staff endpoint trips the auth review's deferred condition:

- **D-Q2** locked verification as a **pluggable adapter, operator-assisted at launch, NIDA later** ‚Äî the **Moderator verification queue is the MVP**, with no launch dependency on NIDA.
- **D15** requires **one account per person** ‚Äî a government-ID submit must dedup **without decrypting** any ID (blind index), and reject a second account on the same ID.
- **D13 / ¬ß25.4** make `isElectoral` **voter-ID-authoritative** with a **change cooldown**; binding civic weight hinges on this single field, so verifying a voter ID must set it authoritatively and demote any manual choice.
- **D16** forbids self-action ‚Äî a Moderator must not approve **their own** verification.
- **N-4 (auth review, blocking):** staff TOTP must be enforced **before any staff-role login path is exposed**. This increment exposes the Moderator endpoint ‚Äî the **first scoped staff endpoint** ‚Äî so staff TOTP cannot be deferred again.
- **MF-2** means tier must remain **server-authoritative**: approval flips a DB flag and the live resolver does the rest; no endpoint may set or trust a client tier.

This ADR decides how the increment earns T2/T3 and lands the first staff surface while honouring every locked decision and PDPA.

## Decision
1. **Profile ‚Üí T2.** Completing names + ‚â•1 `ProfileLocation` + a verified contact channel reaches T2. Contact-channel verify **reuses the existing OTP machinery** with `channel=EMAIL` (and SMS already at signup) ‚Äî zero new OTP logic. The T2 predicate stays owned by `TierService` (client cannot self-assert, MF-2).
2. **ID/voter submit ‚Üí `VerificationRequest(ID, PENDING)`.** A **T2** citizen submits `idType + idNo + fullName + evidenceRef`. The service **field-encrypts `idNo`**, computes the **blind-index `idHash`** and **rejects a duplicate** (`DUPLICATE_IDENTITY`, with the DB unique index as the race backstop), then creates a `PENDING` request through the `IdentityVerificationProvider` **port**. Submitting does not change tier (a pending review never grants/loses tier ‚Äî PRD ¬ß21/¬ß25.5).
3. **Pluggable provider, operator-assisted default (D-Q2).** The `IdentityVerificationProvider` port gets two config-selected adapters: **operator-assisted** (default; returns `PENDING_REVIEW` ‚Üí the Moderator queue) and an **auto-stub** (dev/E2E; the seam for the future NIDA adapter, which returns `PENDING_REVIEW` on outage so the citizen never loses an earned tier). Any provider fault degrades to the queue, never a hard fail (DI2).
4. **Operator (Moderator) queue + approve/reject ‚Äî the first scoped staff endpoint.** Gated on `ROLE_MODERATOR` **and** `@mfa.isStaffMfaSatisfied()` (N-4) **and** `ScopeGuard.canActOnArea` (the subject's ward, effective-window-aware per N-2) **and** `isNotSelf` (D16). **Approve** ‚Üí `profile.idVerified=true` ‚Üí the **live** resolver returns **T3** (no manual `setTrustTier`, MF-2); **reject** ‚Üí reason-coded, tier untouched.
5. **Voter-ID authoritative electoral (D13/¬ß25.4).** On approving a **voter** ID, the resolved registered **ward‚Üíconstituency** is set as the **authoritative `isElectoral`** location ‚Äî auto-creating an electoral-only `ProfileLocation` if that ward is not already pinned, demoting any prior electoral, **bypassing** the cooldown. Residence stays `isPrimary` (voter-ID wins the conflict branch).
6. **Electoral-location management (D13).** Exactly one `isPrimary` and one `isElectoral` per profile (DB partial-unique indexes are the hard backstop). **Manual** `isElectoral` change is **cooldown-guarded** (default **6 months**, config) measured from the durable `electoral_changed_at` column; voter-ID verification bypasses it; re-delimitation re-resolves via the effective-dated `WardConstituency`, with historical actions staying attributed to the constituency in effect when taken.
7. **Staff TOTP enforcement (N-4).** Any account holding `MODERATOR`/`ADMIN`/`ROOT` (or with `mfaEnabled`) must complete a **TOTP second factor**: password/OTP login returns a short-lived `MFA_CHALLENGE` token (new `TokenType`, no roles/tier), and only `/auth/login/totp` with a valid TOTP issues the real pair. A staff account without active MFA **cannot reach the staff endpoint** (`@mfa.isStaffMfaSatisfied()` deny-by-default). TOTP secrets are stored **encrypted** (`CryptoPort`) in two new `app_user` columns (`mfa_pending_secret`, `mfa_totp_secret`).
8. **Audit.** Verification submitted/approved/rejected, tier change, electoral change, and staff-login-with-MFA are appended to the immutable `audit_event` store with **references/hashes only** ‚Äî never `idNo`, raw phone, OTP, or token. New `AuditEventType` values: `AUTH_VERIFICATION_APPROVED`, `AUTH_VERIFICATION_REJECTED`, `ELECTORAL_CHANGED`, `AUTH_LOGIN_MFA`, `AUTH_MFA_ENROLLED`, `AUTH_MFA_CHALLENGE_FAILED`.
9. **Data additions.** Only **two encrypted nullable columns on `app_user`** (`mfa_totp_secret`, `mfa_pending_secret`) in `V6__identity_mfa.sql`; everything else (`VerificationRequest`, `Profile.idVerified/idHash/idType/idNo`, `ProfileLocation.is_electoral/electoral_changed_at`, `OtpChannel.EMAIL`) **already exists**. Plus the `MFA_CHALLENGE` `TokenType` and the new audit enum values.

## Consequences
- (+) Earns T2/T3 **server-authoritatively**: approval flips one DB flag and the live `TierService` does the rest, so the forged-claim defence (MF-2, already tested) extends to verification for free.
- (+) **No launch dependency on NIDA** (D-Q2): the operator-assisted queue is the whole MVP; the auto/NIDA adapter slots behind the same port with the tier predicates unchanged.
- (+) **One account per person holds** under concurrency: blind-index dedup + the `ux_profile_id_hash` unique index reject a duplicate identity even on a race, without ever decrypting an ID (D15, PDPA).
- (+) **Electoral integrity is enforced where it matters**: voter-ID authority + cooldown + single-electoral invariant close the location-hopping attack (R18); the cooldown reads a durable column, so it survives a Redis flush.
- (+) **N-4 closed before the staff door opens**: the Moderator endpoint cannot be reached without TOTP, via both a login gate and an endpoint gate (defence in depth); the platform's first staff surface ships secure.
- (+) **Minimal schema churn**: two encrypted columns; the auth increment's foresight (every other column already modelled) keeps this a near-pure-behaviour increment.
- (‚àí) The Moderator queue is **operator-staffed** ‚Äî throughput is a staffing/program risk (R32), not a code risk; mitigated by scope-partitioning the queue (area scope) and the future auto adapter. Flagged to Program.
- (‚àí) `evidenceRef` depends on the (deferred) object-store + malware-scan increment (EI-8); until it lands the operator works the evidence out-of-band. Accepted: the queue/decision flow is complete around the ref and does not block.
- (‚àí) TOTP adds an enrolment step and a second login round-trip for staff; accepted ‚Äî it is the N-4 security requirement, and citizens are unaffected (TOTP only gates staff roles).
- (‚àí) Email-channel OTP rides the SMS stub seam until the real `EmailSender` adapter lands; accepted (dev/test parity, no external call).
- **Revisit triggers:** (a) **NIDA/voter-ID API access granted** ‚Üí swap the operator-assisted adapter for the registry API behind the same port (D-Q2), Moderator queue becomes the degradation/fallback only; (b) **`REP_CLAIM`/`ORG` verification volume** ‚Üí the typed queue already supports them; add their decision rules + scope without a new surface; (c) **MFA factors beyond TOTP** (WebAuthn/passkeys for staff) ‚Üí promote the two `app_user` secret columns to a `user_mfa` table (the ADR-0011 trigger, now concrete); (d) **cooldown/electoral-policy tuning** is config-driven (`cooldown-days`) ‚Äî a policy change needs no migration; (e) **fraud-driven revocation recompute** (¬ß25.5) becomes load-bearing ‚Üí owned by `engagement`/`accountability`, re-validating one-per-person after aggregate recompute.
